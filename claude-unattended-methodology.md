# Claude Code 无人值守 — 分析链路与推导方法论

> 本文档记录从"想实现无人值守"到"六层防御体系完整方案"的完整推导过程。
> 重思路、重推导链路、重组网方式——而非重复具体方案内容。

---

## 一、起点：问题定义（不是"我要什么"，是"什么在阻止我"）

### 1.1 初始问题

用户输入：「如何实现 Claude 在任务制定后的无人值守，避免用户必须常驻确认操作」

### 1.2 问题分解的方法

**不要直接跳到"答案"。先问：什么在阻止无人值守？**

```
用户想达成: 设定任务后离开，Claude 自己完成
实际发生的: Claude 停下来等用户点"允许"
```

从现象逆推根因：

```
现象 1: 弹窗要求确认工具调用
  → 根因: 权限模型默认交互式
  → 解决域: settings.json + --permission-mode

现象 2: 终端关了 Claude 就没了
  → 根因: 进程绑在用户 session 上
  → 解决域: daemon / systemd / tmux

现象 3: 不知道什么时候让 Claude 做事
  → 根因: 缺少调度机制
  → 解决域: cron / CronCreate / /loop

现象 4: 做完任务不知道结果
  → 根因: 缺少通知通道
  → 解决域: PushNotification / webhook

现象 5: 网络中断后任务全废
  → 根因: 任务状态只在 Claude 的内存里
  → 解决域: 外部化 + 恢复协议
```

**关键方法论：每个"现象"映射到一个"根因"，每个根因映射到一个"解决域"。不跳过现象直接讨论方案。**

### 1.3 层次化的核心洞察

在分析过程中，逐渐浮现出一个关键洞察：

**无人值守不是单一技术点，而是正交的多个层次。** 每个层次独立决策，但必须全部覆盖。

这个洞察来自于发现：
- 权限配置好了，但终端关了还是不行 → 持久化是独立层
- 持久化解决了，但任务什么时候触发 → 调度是独立层
- 调度解决了，但网络断了怎么办 → 中断恢复是独立层
- 中断恢复了，但 Claude 思路跑偏了 → 语境连续是独立层

**方法论：当你发现一个"方案"解决了问题A但问题B还在，那问题B就是下一个独立层。**

## 二、分析推进逻辑（从现象到方案的思维链）

### 2.1 Layer 1 (权限) 的推导

```
为什么弹窗？→ 因为 Claude 需要确认每个工具调用
怎么解决？→ 告诉 Claude "这个项目的这些命令不需要确认"
怎么告诉？→ settings.json 的 allow/deny 清单
怎么配？→ allow 用命令前缀匹配（Bash(git:*)），deny 用破坏性模式（Bash(rm -rf /)）
怎么验证够了？→ claude -p "ok" --permission-mode accept-edits 无交互执行
```

**推导中犯过的错误**：一开始想全局 bypass。后来意识到——bypass 只适合 CI/CD 环境，日常开发需要精细控制。**方案的选择不是"哪个最好"，是"哪个匹配当前场景的风险容忍度"。**

### 2.2 Layer 2 (持久化) 的推导

```
Claude 为什么终端关了就没？→ 因为它是前台进程，绑在 bash session 上
怎么让它不绑？→ 两种思路：A) session 管理工具 (tmux)  B) 系统服务管理 (systemd/launchd)
当前环境有哪个？→ 都没有！tmux 没装，systemd 不存在（PRoot）
那怎么办？→ Claude 自带 daemon 模式，PID 挂在 init 下，不依赖终端
daemon 稳定吗？→ 查日志：经历过多次崩溃和自动恢复，二进制更新后也会自动重启
```

**推导中的关键拐点**：发现环境中没有 systemd 也没有 tmux，差点以为"无解"。但仔细审视发现 Claude 自己就有 daemon 模式，且已经在跑（PID 21625）。**不要假设"缺了什么"，要先检查"已经有什么"。**

### 2.3 Layer 5 (中断恢复) 的推导

```
问题: 网络断了，任务白做
为什么白做？→ 因为 Claude 把"做到哪了"只记在上下文里，断了就没了
解决方向？→ 在 Claude 的上下文之外，用一个文件记录进度
什么格式？→ 先用最简单的：一个日志文件，每步追加一行
够不够？→ 不够。日志告诉你做到第几步，但不告诉你是"怎么做"的
还缺什么？→ 决策理由、代码理解、已发现的要点
→ 这就是 context-dump.md 的由来
```

**推导中的关键拐点**：从 task-state.json（WHAT）到 context-dump.md（WHY），是两个不同层次的问题。前者解决"进度"，后者解决"思路"。**每当你觉得"方案好像还不够"，问自己：是"不够详细"还是"不在同一层"？**

### 2.4 Layer 6 (Socket 错误消除) 的推导——最完整的链路

这是整个分析中最深的一层。推导过程：

```
Step 1: 精确定义问题
  错误信息: "The socket connection was closed unexpectedly"
  不是 "request failed"，不是 "timeout"，是 "closed unexpectedly"
  → 这意味着 TCP 连接被静默关闭了

Step 2: 理解发生的精确时间线
  T+0: Claude 发 API 请求 → 建立连接
  T+1-10s: 流式响应
  T+10-60s: Claude 执行工具，连接空闲
  T+60s: Claude 发下一个 API 请求 → 发现连接断了
  → 问题出在"空闲期"！不是请求期间，是请求之间

Step 3: 画网络路径
  Claude (Node.js fetch) → 内核 TCP → PRoot → Android → NAT → 互联网 → DeepSeek
  每一层都可能切断空闲连接

Step 4: 逐层排查空闲超时
  移动运营商 NAT: 30-120s（不可控）
  Linux TCP keepalive: 默认 7200s → 远超过 NAT 的 30s（可控！）
  Node.js HTTP Agent keepAlive: 默认 false → 每个请求新连接（可控！）
  Cloudflare: ~100s（不可控）
  DeepSeek LB: ~60-120s（不可控）

Step 5: 关键发现
  - 内核 keepalive=7200s，NAT 超时=30-120s → keepalive 形同虚设
  - Node.js 不复用连接 → 即使复用，空闲时也不发心跳
  → 两个可控因素都没起到作用

Step 6: 解决方案设计
  找到回答"可控的那两个"：
  A) sysctl keepalive=60s → 内核每 60s 发探测，NAT 就不会超时
  B) 透明代理 → 在应用层加连接池和心跳，socket 错误自动重试
  不可控的部分用"容忍"覆盖：
  C) 如果 A+B 都失败了（基站物理切换）→ L5 中断恢复兜底
```

**这个推导链路的精髓**：从"报错信息"→"时间线"→"网络路径"→"逐层超时"→"可控vs不可控"→"分层防御"。**不是从"有什么方案"出发，是从"根因链"出发。**

## 三、重组网方式（如何将零散发现组合成体系）

### 3.1 发现驱动的层次构建

整个六层框架不是预先设计的，是**被问题逼出来的**：

```
最初: 只需要 Layer 1 (权限)
  用户追问: "终端关了怎么办" → Layer 2 (持久化)
  用户追问: "怎么定时执行" → Layer 3 (调度)
  用户追问: "怎么知道结果" → Layer 4 (通知)
  用户追问: "网络断了怎么办" → Layer 5 (中断恢复)
  用户追问: "恢复后思路跑偏" → Layer 5.5 (语境连续性)
  用户追问: "能不能让错误不发生" → Layer 6 (错误消除)
```

**方法论：层次是被追问出来的，不是设计出来的。** 用户的每个追问暴露了当前方案的一个盲区，盲区补上就是一个新层。

### 3.2 平台无关性的抽象

在 Layer 2 (持久化) 的分析中，发现不同平台有不同的进程托管方式：

```
Linux → systemd
macOS → launchd
Windows → Task Scheduler
Docker → restart policy
```

关键洞察：**要抽象的是"需要什么能力"，而非"用什么工具"。**

```
需求抽象: "我需要一个东西，在进程死掉后自动拉起来"
  Linux 实现: systemd Restart=on-failure
  Docker 实现: restart: unless-stopped
  macOS 实现: launchd KeepAlive

需求抽象: "我需要一个东西，在固定时间点触发任务"
  Linux 实现: cron
  macOS 实现: launchd StartCalendarInterval
  CI/CD 实现: GitHub Actions schedule
```

**方法论：先定义"我需要什么能力"，再找"当前平台怎么实现这个能力"。不要反着来。**

### 3.3 方案粒度决策

整个分析过程中，做了多次粒度决策：

| 层次 | 粒度决策 | 理由 |
|------|----------|------|
| 权限 | allow 用前缀匹配而非精确匹配 | 精确匹配导致换个仓库就弹窗 |
| 任务状态 | task-state.json 之外再加 context-dump.md | task-state 只管进度，不管思路 |
| 恢复 Prompt | "你不是在开始新任务"在前 | 先建立心理锚点，再给具体指令 |
| Socket 消除 | 代理而非修改 Claude 源码 | 代理对 Claude 透明、对 DeepSeek/Anthropic 都有效 |

**方法论：粒度选择的依据是"这个方案会被用在多少种不同场景下"。场景越多，粒度越粗（越通用）；场景越固定，粒度越细（越高效）。**

## 四、从分析到文档的转化方法论

### 4.1 每篇文档的结构模式

观察输出的 5 篇核心文档，它们的共同结构：

```
1. 问题定义   — "什么现象，什么根因"
2. 环境分析   — "当前有什么约束，有什么能力"
3. 方案总览   — "整体架构图/层次图"
4. 逐层展开   — "每层的 WHY-WHAT-HOW"
5. 立即实施   — "最小可行步骤"
6. 效果量化   — "改前 vs 改后"
7. 总结       — "一句话核心原则"
```

### 4.2 文档间的正交性

```
无人值守方案（环境专属） ←→ 跨平台通用指南
  具体环境的实施清单          抽象能力和选型框架

中断恢复                    ←→ 语境连续性
  WHERE: 做到哪了            WHY: 为什么这样做
  
中断恢复 + 语境连续性       ←→ Socket 错误消除
  发生了怎么办               怎么让错误不发生
```

**方法论：确保每篇文档有独立的"回答域"。如果两篇文档回答同一类问题→合并；如果回答不同维度的问题→保留但交叉引用。**

## 五、核心方法总结

### 5.1 分析原则

| 原则 | 含义 | 违反后果 |
|------|------|----------|
| **从现象到根因** | 不要直接跳到方案，先画现象→根因链 | 治标不治本 |
| **逐层排查** | 网络问题逐层（应用→TCP→NAT→路由→服务端）排查超时 | 改了不该改的层 |
| **可控 vs 不可控** | 区分哪些因素你能改，哪些不能 | 在不可控因素上浪费精力 |
| **先勘察后设计** | 搞清环境有什么/缺什么，再设计方案 | 方案无法落地 |
| **被追问驱动演进** | 层次不是预先设计的，是被用户的追问逼出来的 | 过度设计 |

### 5.2 设计原则

| 原则 | 含义 |
|------|------|
| **正交分层** | 每层解决一个独立问题，层之间不耦合 |
| **能力抽象** | 先定义"需要什么能力"，再找"平台怎么实现" |
| **粒度匹配场景** | 场景越多粒越粗，越固定粒越细 |
| **外部化状态** | 任何需要恢复的东西，不能只存在 Claude 的内存里 |
| **防御深度** | 不让错误发生 > 发生了自动重试 > 重试失败精准恢复 |

### 5.3 输出原则

| 原则 | 含义 |
|------|------|
| **金字塔结构** | 结论在前，推导在后，细节在最后 |
| **可立即执行** | 每篇文档都有"立即可做的三步" |
| **量化对比** | 用 token 数、时间、概率衡量方案效果 |
| **一句总结** | 每篇文档标题下有一句话核心原则 |
