# 最优韧性设计 — 四目标约束下的决策推导

> 四个硬约束，一个都不能偏：
> ① 最大限度节省 token ② 保证输出质量和一致性 ③ 尽量中断无感 ④ 目标不偏移
> 本文记录每一个设计决策如何在这四个约束下做出。

---

## 一、四约束之间的天然矛盾

```
节省 token ←→ 中断无感
  │              │
  └── 门控等待   ──→ 增加了延迟, 但用户可能感知到停顿
  └── 少重试     ──→ token 省了, 但更容易让错误到达 Claude
  └── 关 thinking ──→ token 大省, 但 → 违反「输出质量」
  
输出质量 ←→ 节省 token
  │           │
  └── thinking ──→ 质量提升, token 消耗大, 中断后浪费严重
  └── 长上下文  ──→ 质量提升, input token 大, 重试代价大

中断无感 ←→ 输出质量
  │           │
  └── 快速重试  ──→ 恢复快, 但可能还在网络抖动中 → 再次失败
  └── 降级模式  ──→ 去掉 thinking 快速返回, 但质量下降
```

**核心矛盾：不可能三个同时拿到极致。必须有优先级。**

---

## 二、优先级排序（设计基石）

```
在发生冲突时, 按此顺序做取舍:

1. 输出质量 > 节省 token
   → 不能用低质量方案换 token 节省
   → 不能用 cheap model 代替 thinking model
   → 不能截断上下文

2. 节省 token > 中断无感
   → 宁可等 15 秒让网络稳定, 也不冒险发请求浪费几千 tokens
   → 宁可让用户感知到短暂停顿, 也不在抖动时盲目发送

3. 中断无感 > 零延迟
   → 接受数秒延迟, 换取自动恢复
   → 不接受「需要用户输入关键字」
```

**优先级的后果：**
- 门控等待是首选策略（省 token > 延迟）
- thinking 绝不能降级（质量 > 一切）
- 代理必须透明（无感 > 零延迟）

## 三、每个设计决策的约束校验

### 决策 1: 稳定性门控（等网络稳了再发）

```
对 ① 节省 token 的影响: ✅✅ 最大贡献者
  避免「发了请求→网络中途断→服务器已处理并计费→浪费」
  每次避免浪费: X input + 部分 thinking tokens (几百~几千 tokens)
  
对 ② 输出质量的影响: ✅ 零影响
  请求本身不变, 只是延迟发送。服务器收到的请求一模一样。
  
对 ③ 中断无感的影响: ⚠️ 轻微负面
  网络切换后多等 5-15s, Claude 看起来"卡了一下"
  但不会报错、不会停止、不需要用户输入 → 仍然是无感范畴
  
对 ④ 目标偏移: 不偏移
  门控服务于「省 token」, 副作用是短暂的额外延迟
  延迟上限 90s → 不会无限等
```

**结论: 保留, 且是核心策略。**

### 决策 2: thinking 请求使用更高的稳定性阈值

```
对 ① 节省 token 的影响: ✅✅ 
  thinking tokens 按 output 计费, 中断后浪费最严重
  提高门槛 = 减少 thinking 中断概率 = 直接省大额 tokens
  
对 ② 输出质量的影响: ✅ 零影响
  不改变 thinking 行为, 只在发送时机上更谨慎
  
对 ③ 中断无感的影响: ⚠️
  thinking 请求的门槛 0.9 vs 普通请求 0.8
  → thinking 请求更可能被门控 → 多等 5-15 秒
  → 但 thinking 本身就是重操作, 用户对该操作的延迟预期更高
```

**结论: 保留。thinking 请求的延迟换取确定性, 值得。**

### 决策 3: 重试时保持相同请求参数（不降级 thinking）

```
对 ① 节省 token 的影响: ❌ 负面
  重试不关 thinking → thinking 从头重新生成 → 多消耗 thinking tokens
  
  备选方案: 重试关 thinking
  → 省了 thinking tokens
  → 但 ↘ 违反约束②（输出质量）
  
对 ② 输出质量的影响: ✅ 
  不会因为网络问题而降低输出质量
  
对 ③ 中断无感的影响: 无关
```

**结论: 不降级。质量 > 省 token。**

### 决策 4: HEAD 预检 — 发请求前确认服务器可达

```
对 ① 节省 token 的影响: ✅ 
  如果服务器不可达 → 不发送请求 → 零 token 浪费
  HEAD 请求本身开销: TCP+TLS 握手 (~1KB 流量, 0 tokens)
  
对 ② 输出质量的影响: ✅ 零影响
  
对 ③ 中断无感的影响: ✅ 正面
  减少"发出去才发现不通" → 减少报错 → 更无感
```

**结论: 保留。近乎零成本的正收益。**

### 决策 5: 代理缓冲响应 → 全部收到后再转发给 Claude

```
对 ① 节省 token 的影响: ✅ 
  如果响应传输中断 → Claude 没收到任何东西 → 不知道发生过错误
  → 历史中没有报错记录 → 重试时 input 不膨胀
  
  如果用流式转发:
  → Claude 收到部分响应 → Claude 可能会处理不完整的信息
  → 报错后, 历史中记录了"部分响应 + 错误" → 重试时 input 多出这部分
  
对 ② 输出质量的影响: ⚠️ 
  失去了流式体验（逐字输出）
  但最终结果完全一致
  
对 ③ 中断无感的影响: ✅ 正面
  Claude 不会收到半截响应 → 不会进入不一致状态
```

**结论: 保留。牺牲流式体验, 换取 token 节省和一致性。**

### 决策 6: 不启用守护/--resume（v2 修正）

```
对 ① 节省 token 的影响: 无影响
  
对 ② 输出质量的影响: ✅
  --resume 不恢复对话上下文 → 可能导致新会话用不同思路
  → 不依赖 --resume 而用新会话+恢复 prompt → 更可控
  
对 ③ 中断无感的影响: ⚠️
  代理耗尽→错误到达 Claude→Claude 停止等输入→需要用户说"继续"
  但这个场景（代理四道防线全部失败）概率极低
  且如果发生 → Claude 的会话历史完整保留 → 用户说"继续"后质量不受影响
```

**结论: 不部署守护。代理四道防线已覆盖 99%+ 网络中断场景。剩余 1% 留给用户手动"继续"——此时 token 开销和一致性都不受影响。**

### 决策 7: 不缓存/不复用之前的部分响应

```
备选方案: 代理缓存部分响应, 中断后重试时尝试复用
  
  为什么不行:
  → 部分 thinking + 部分 text → 不是有效的 API 响应
  → API 不支持"从中间续传"
  → 强行拼接会产生幻觉
  
对 ① 节省 token: 有诱惑, 但不可行
对 ② 输出质量: ❌ 强行拼接破坏质量
```

**结论: 不缓存。API 协议不支持, 不能为了省 token 牺牲正确性。**

## 四、最优参数配置

基于四约束权衡后的最终参数：

| 参数 | 值 | 约束依据 |
|------|-----|---------|
| 重试次数 | 3 | 平衡: 太少→错误到达Claude, 太多→浪费token |
| 重试间隔 | 1s / 3s / 8s | 指数回退, 8s上限防过度等待 |
| 普通请求稳定性阈值 | 0.8 | 80%成功率即可发送, 不追求完美 |
| thinking 请求稳定性阈值 | 0.9 | 提高20%, 避免最贵的浪费 |
| thinking 确认连胜次数 | 5 | 比普通请求多2次确认, 对抖动更敏感 |
| 最大挂起时间 | 90s | 留给Claude 30-90s 处理请求 |
| 确认延迟 | 2s | 连续两次探测都成功才算稳 |
| 探测间隔 | 5s | 5秒一次HEAD, 快速感知恢复 |
| 连接保活间隔 | 45s | 低于 NAT 60s 超时, 留15s缓冲 |
| TCP keepalive | 60s | 低于 NAT 120s, 留60s缓冲 |

## 五、不可触碰的底线（防止目标偏移）

```
硬底线 1: 不降级 thinking
  任何情况下都不能因为网络问题关闭 thinking 或用 cheaper model
  这直接违反「保证输出质量」

硬底线 2: 不截断上下文  
  重试请求必须包含完整的对话历史
  不能因为节省 input token 而删除历史消息
  
硬底线 3: 不伪造响应
  不能缓存部分响应拼接后返回给 Claude
  代理不能捏造任何 API 行为

硬底线 4: 不无限等待
  门控必须有时限 (90s)
  超时后或因发送(接受风险)或因返回错误(让用户决定)
  
硬底线 5: 不对Claude隐藏真实的失败
  如果四道防线全部失败, 代理必须如实返回错误
  不能让 Claude 认为"请求成功了", 实际没有
```

## 六、最终架构总结

```
请求到达代理
     │
     ▼
┌─────────────┐
│ L0 门控      │ 网络不稳? → 挂起,等稳了再过
│   检查稳定性  │ thinking? → 门槛更高 (0.9 vs 0.8)
└──────┬──────┘
       │ 通过
       ▼
┌─────────────┐
│ L1 预检      │ HEAD 确认服务器可达
│   快速探测   │ 不通 → 标记失败, 重试
└──────┬──────┘
       │ 通过
       ▼
┌─────────────┐
│ L2 发送      │ 使用连接池 (keepalive + heartbeat)
│   (含重试)   │ 完整请求体, 不改参数
└──────┬──────┘
       │ 成功
       ▼
┌─────────────┐
│ 缓冲完整响应 │ 不流式转发, 确保完整性
│ 再转发Claude │ Claude 只看到一次成功响应
└─────────────┘

四条防线全失败 → 错误到达 Claude → Claude 停止, 等用户输入"继续"
(此时代理会话历史完整, 不会重复计费, 输出质量不受影响)
```

## 七、效果预估

```
场景分布 (移动网络, 24小时):
├── 95% 网络持续稳定     → 代理零开销, 零延迟, token正常
├──  3% 瞬时抖动 (<12s)  → 代理重试成功, 0-12s延迟, 0额外token
├──  1% 短断网 (12-90s)  → 门控挂起, 最多90s延迟, 0额外token  
├──  0.5% 长断网 (>90s)  → 门控超时, 或发或报错, 可能1次额外token
└──  0.5% 物理断连       → 错误到达Claude, 用户手动"继续", 0额外token

额外 token 开销: 从「每次抖动可能浪费几百~几千 tokens」
               降为「仅长断网场景可能有 1 次额外 input tokens」
               降幅 > 95%
               
质量影响: 零（不降级 thinking, 不截断, 不伪造）

延迟增加: 网络切换后额外 5-15s, 网络稳定时 0
```
