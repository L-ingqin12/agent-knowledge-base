# 上下文工程：为什么窗口越大模型越蠢

## 摘要

Transformer 的注意力是有限且需要分配的——context 每多一个 token，真正关键信息的注意力份额就被稀释一分。这是「窗口 100 万但回答更蠢」的根源，由此衍生三个具体现象：**注意力稀释**（噪声淹没信号）、**Lost in the Middle**（context 中间段信息命中率断崖下跌，呈 U 形曲线）、**Context Rot**（长 Agent 轨迹中过期矛盾信息堆积，模型反刍自己制造的垃圾）。

工程解法四层叠加：**检索**（只放相关片段，性价比最高）、**压缩**（长轨迹摘要，留结论留约束留待办，扔过程噪声）、**子 Agent 隔离**（独立 context 互不污染）、**渐进式披露**（Skills 式按需分级加载）。核心原则：**哪怕窗口有 100 万 token，也要像只有 8000 一样经营 context——永远只放信号，不放噪声。** 有效上下文 > 最大上下文。

---

## 一、核心认知：Context 不是仓库，是注意力预算

Transformer 每生成一个 token，都要在所有已有 token 上做注意力计算。注意力的总量是有限的，在所有 token 之间**此消彼长竞争**。

| Context 长度 | 每条关键信息分到的注意力 |
|-------------|---------------------|
| 1000 token | 充裕 |
| 50 万 token | 被几十万个不相关 token 稀释到几乎可忽略 |

类比：context window 是会议室的桌子，不是隔壁的档案室。档案室能放一万份文件（训练知识 + 外部知识库），但会议桌上真正摊开讨论的只能是有限的几份。全堆上去的结果不是「信息更全」，而是「谁也找不到要讨论的那份」。

---

## 二、三个具体现象

### 2.1 注意力稀释

塞满 100 万 token ≠ 这 100 万 token 都在为当前回答出力。大量 token 只是在**互相干扰**。一条明明在 context 里的信息被记错——不是模型没「存」住，是它的注意力权重被几千条噪声冲没了。

### 2.2 Lost in the Middle（中间盲区）

斯坦福研究：同一条关键信息分别放在长文档的**开头、中间、结尾**，问相关问题。结果呈 U 形曲线：

| 信息位置 | 答对率 |
|---------|--------|
| 最开头 | 最高 |
| 最结尾 | 次高 |
| 正中间 | 断崖式下跌，部分设置下接近「根本没给这条信息」 |

模型读长 context 跟人考前抱佛脚一样——首因效应 + 近因效应，记头记尾，中间大段划水。

**工程杀伤力**：彻底否定「窗口大就按文档原始顺序一股脑灌」。文档物理顺序和信息重要性是两件无关的事。重要信息落在中间区 = 等于没放。

**实测案例**（金融保险 RAG 项目）：检索回 20 个 chunk 按相似度顺拼，Top1 chunk 命中了答案但模型用了排第 8 的错误 chunk。原因——最关键的几个 chunk 挤在 context 中段盲区。改为最相关 chunk 放首尾、次要放中间，同一套检索结果，答案正确率直接抬一截。

### 2.3 Context Rot（上下文腐烂）

长 Agent 轨迹（几十步 tool call）中堆积大量过期、矛盾、自我引用的内容，模型输出持续退化且「越陷越深」。

根源：过期信息占用注意力预算（稀释），且这些信息体量大、反复出现，权重不降反升（位置/频次）。

典型场景：Agent 前面试了方案 A 失败（全写进 context）、方案 B 也失败（全写进 context），走到第 30 步，context 里全是「我刚才搞砸了」的痕迹——模型不断重复已证明无效的思路（在 context 里反复出现，注意力权重高），或彻底迷失在一堆中间状态里。

关键认知：不能指望模型「更聪明地忽略垃圾」——没法跟概率模型讲「这段你别当真」。必须从工程上**阻止垃圾在 context 里堆积**。

## 三、四层工程解法（从轻到重，叠加使用）

| 层级 | 解法 | 原理 | 典型实现 |
|------|------|------|----------|
| 1 | **检索**（Retrieval） | 5000 份文档不灌，先检索出当前问题真正需要的 3-5 个片段 | RAG |
| 2 | **压缩**（Compaction） | 20 步探索过程 → 结构化进展摘要（做过什么、结论、卡在哪）→ 几百字替换几万 token 原始轨迹 | Claude Code context 自动压缩 |
| 3 | **子 Agent 隔离** | 大任务拆给多子 Agent，各自独立干净 context，只把结论回传主线 | 多 Agent 架构 |
| 4 | **渐进式披露** | 第一级只放一句话元信息（让模型知道有这个能力），真正用时才加载第二级完整说明，需要文件才读第三级 | Claude Skills 分级加载 |

### 3.1 检索——性价比最高的第一步

RAG 的深层价值不只是「让模型访问外部知识」，更是**「让模型不必访问无关知识」**。把 100 万 token 噪声压成几千 token 信号。

### 3.2 压缩——长了就缩

不能简单砍掉最早几步（可能砍掉关键约束）。压缩的艺术：**留结论、留约束、留待办，扔掉过程噪声**。

### 3.3 子 Agent 隔离——分而治之

表面是「分工」，底层是「context 隔离」。子 Agent 干完活只回传结论，中间探索过程烂在自己 context 里，不污染主线。

### 3.4 渐进式披露——按需加载

Claude Skills 的核心设计。背后无论挂多少知识，任何时刻 context 里只有当前这一步真正用得上的部分。真实系统里四层叠用：检索取相关 → 压缩控长度 → 隔离防污染 → 渐进式减冗余。

---

## 四、核心原则

**有效上下文 > 最大上下文**。长窗口是双刃剑——它给了你「可以偷懒全塞进去」的能力，而这恰恰是大多数人翻车的根源。

成熟做法：哪怕有 100 万 token 窗口，也像只有 8000 token 一样经营 context。每放一段进去之前先问——**这段对当前这一步的回答，是信号还是噪声？**

---

## 五、面试应答框架

被问「为什么 context 越长模型表现反而下降」时：

1. **破误区（30s）**：上下文窗口不是仓库而是注意力预算。Transformer 注意力有限且竞争分配，context 越长 → 关键信息注意力份额被稀释 → 噪声淹没信号。

2. **位置效应（30s）**：Lost in the Middle。同一条信息放 context 首尾命中率高，放中间断崖下跌呈 U 形曲线。不能按文档原始顺序硬灌，最关键信息必须放首尾位置。

3. **长程退化（20s）**：Context Rot。多轮/长 Agent 轨迹中过期矛盾信息堆积，带着模型重复无效路径。

4. **工程解法（40s）**：四层叠加——检索只放相关片段、压缩做轨迹摘要、子 Agent 隔离独立 context、渐进式披露按需加载。核心原则：窗口 100 万也要像 8000 一样经营，只放信号不放噪声。
